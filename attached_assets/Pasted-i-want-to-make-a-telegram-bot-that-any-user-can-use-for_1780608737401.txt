Where the Web App (Telegram Mini App) Fits
For your bot, the Mini App (the UI that opens inside Telegram) is not first and not last — it's a parallel layer you build on top of working bot logic.

Here's the thinking:

You can't build the UI before the backend exists. The Mini App is just a frontend — it calls your bot's functions. If those functions aren't coded yet, the UI has nothing to connect to.

You don't want to leave it to last either. Some features like multi-select chat lists, premium dashboards, and duplicate logs are genuinely painful to do through bot commands alone. Building the UI too late means you coded command-based versions of things you'll just redo.

The Practical Answer
Build the Mini App incrementally starting after Part 2, alongside the other parts:

When	What to build in the Mini App
After Part 2	Login flow, chat list with multi-select
After Part 3	Forwarding settings, source/destination picker
After Part 4	Duplicate logs viewer
After Part 5	Join request settings, Q&A editor
After Part 6	Moderation panel, ban list manager
After Part 8	Premium status, upgrade screen
On Render, You'll Create
Service	Type
Bot backend (Python/Telethon)	Web Service
Job queue worker	Background Worker
Scheduled tasks (keep-alive, timers)	Cron Job
Mini App frontend (HTML/JS)	Static Site
Redis	Key Value
Database	MongoDB Atlas (external, free)
So you'll actually be using 4–5 Render service types from that screen — the Static Site is specifically your Mini App frontend.
